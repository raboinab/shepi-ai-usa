## Goal

Turn shepi into an OpenAI ChatGPT App: a discoverable, user-authenticated MCP server that exposes the existing shepi tools and renders interactive widgets inside ChatGPT for key actions like viewing a project summary or QoE metrics.

## Why this requires a new MCP server

The current MCP server is generated by `@lovable.dev/mcp-js` (v0.20.0). It handles Supabase OAuth 2.1, protected-resource metadata, and the existing 10 tools, but the Lovable SDK does **not** expose the MCP Apps widget APIs (`registerAppResource`, `registerAppTool`, `RESOURCE_MIME_TYPE` from `@modelcontextprotocol/ext-apps/server`). Those APIs require the official `@modelcontextprotocol/sdk` server.

To avoid breaking the existing Claude / generic-MCP connector flow, I propose adding a **second, ChatGPT-specific MCP Edge Function** (`chatgpt-mcp`) rather than replacing the current `mcp` function.

## Proposed approach

1. **Keep the existing `mcp` function** (`https://<project-ref>.supabase.co/functions/v1/mcp`) as the text-only connector for Claude and other MCP clients.
2. **Create a new `chatgpt-mcp` Edge Function** (`https://<project-ref>.supabase.co/functions/v1/chatgpt-mcp`) using the official MCP SDK + `@modelcontextprotocol/ext-apps/server` for widgets.
3. Reuse the **same Supabase OAuth 2.1 auth** (issuer: `https://<project-ref>.supabase.co/auth/v1`, audience `authenticated`) so users approve once per connection.
4. Reuse the **existing project/QoE calculation logic** by moving a focused server-side adapter into `supabase/functions/_shared/qoe/`. The current `src/lib/projectToDealAdapter.ts` + `src/lib/qoeMetrics.ts` are not shared with Edge Functions; a server-side copy is needed.

## Implementation phases

### Phase 1 — Stabilize current deployment
- Add `mcp` to `supabase/config.toml` with `verify_jwt = false` (matching `alice-mcp`).
- Re-deploy the existing `mcp` function and verify `/.well-known/oauth-protected-resource` and `/.mcp/list-tools` still respond.

### Phase 2 — Server-side QoE adapter
- Create `supabase/functions/_shared/qoe/projectToDealAdapter.ts` and `qoeMetrics.ts` using the existing `_shared/workbook/calculations.ts`, `deal-labels.ts`, and `workbook-types.ts` already present in Edge Functions.
- Keep the adapter focused on what the widgets need: periods, accounts, trial balance, adjustments, reclassifications, addbacks, and LTM QoE metrics.

### Phase 3 — ChatGPT MCP server scaffold
- Install runtime deps: `@modelcontextprotocol/sdk`, `@modelcontextprotocol/ext-apps`, `zod` (already present).
- Create `supabase/functions/chatgpt-mcp/index.ts`:
  - `McpServer` from the official SDK.
  - Supabase OAuth 2.1 bearer verification using the same pattern as the current `mcp` function (verify `iss`, `aud`, `client_id`, forward token to Supabase for RLS).
  - CORS handling and `Accept: application/json, text/event-stream` patching for MCP Streamable HTTP compliance.
  - Health / metadata endpoints.

### Phase 4 — Port tools to ChatGPT MCP server
Port these tools from the Lovable SDK shape to the official SDK shape, preserving titles, descriptions, annotations, and `structuredContent`:
- `echo`
- `list_projects`
- `get_project_summary`
- `list_adjustments`
- `get_quality_of_earnings_summary`
- `update_adjustment_status`
- `create_project`

`get_export_data` and `list_documents` can follow in a second pass.

### Phase 5 — Build the first interactive widget
Create a single-file React widget bundle:
- `src/chatgpt-widgets/project-summary.html` + `src/chatgpt-widgets/project-summary.tsx`
- Uses `@modelcontextprotocol/ext-apps/react` (`useApp`) to receive the tool result and call server tools from the iframe.
- Displays project metadata + LTM QoE metrics (Revenue, Gross Profit, Reported EBITDA, Adjustments, Adjusted EBITDA) using the existing Shepi semantic tokens.
- Build it with Vite + `vite-plugin-singlefile` into a self-contained HTML file.

### Phase 6 — Wire widget into the MCP server
- In `chatgpt-mcp/index.ts`, register `ui://project-summary/mcp-app.html` as a resource with `RESOURCE_MIME_TYPE`.
- Link `get_project_summary` and `get_quality_of_earnings_summary` to that resource via `_meta: { ui: { resourceUri } }`.
- Decide how to serve the HTML inside the Edge Function:
  - **Recommended:** embed the built HTML as a string at build time using a small script that writes `supabase/functions/chatgpt-mcp/widgets/projectSummary.ts` from the Vite output; the function imports that string. This avoids a runtime dependency on the public site being deployed.

### Phase 7 — Update marketing/discovery surfaces
- `src/pages/Connect.tsx`: add a ChatGPT Apps section with the `chatgpt-mcp` URL, developer-mode steps, and a note that Claude users should use the existing `mcp` URL.
- `src/pages/ForAiAgents.tsx`: mention ChatGPT Apps support.
- `public/.well-known/agent.json`: update `capabilities` / `endpoints` to reference the ChatGPT App MCP server.
- `public/mcp.json`: leave as the static content descriptor; consider adding a note about the live MCP servers.
- `scripts/generate-discovery.ts` + `src/App.tsx` + `public/sitemap.xml`: ensure any new route is prerendered and indexed.

### Phase 8 — Test and deploy
- Deploy `chatgpt-mcp` with `supabase--deploy_edge_functions`.
- Add `chatgpt-mcp` to `supabase/config.toml` with `verify_jwt = false`.
- Test with ChatGPT developer mode by creating a connector pointing at `https://mdgmessqbfebrbvjtndz.supabase.co/functions/v1/chatgpt-mcp`.
- Verify:
  - OAuth consent flow works and returns the user to ChatGPT.
  - `list_projects` returns text/structured data.
  - `get_project_summary` renders the widget inline.
  - The widget can call `tools/call` back to the server.
- Run project tests, build, and update `.lovable/mcp/manifest.json` if needed (the Lovable manifest covers the existing `mcp` function; the ChatGPT function does not need a Lovable manifest).

## Risks and open decisions

1. **Transport layer:** The official `@modelcontextprotocol/sdk` `StreamableHTTPServerTransport` is Node/Express-oriented. For Deno/Supabase we may need to use a Web-Standard-compatible transport or a thin adapter. The `mcp-lite` transport used by `alice-mcp` is Deno-friendly but may not be compatible with the official SDK's `McpServer`. This is the first technical spike.
2. **Widget HTML delivery:** Embedding the HTML as a build-time string is the most robust option for Edge Functions; we need a small script to generate the import.
3. **Scope of first widget:** I recommend only the project summary / QoE widget for the first iteration. Additional widgets (adjustment list, document list, create-project form) can follow without changing the server architecture.
4. **Auth parity:** We must verify that the official SDK's OAuth verification accepts the same Supabase tokens as the Lovable SDK (same issuer, audience, `client_id` claim). If not, the ChatGPT server may need `requireOAuthClientClaim: false` or a different token policy.

If you approve this plan, I'll start with Phase 1 (stabilize the existing MCP deployment) and Phase 2 (server-side QoE adapter), then move to the ChatGPT server and first widget.
